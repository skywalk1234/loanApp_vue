# 申请评分卡字段映射报告

## 1. 报告目的

本项目当前模型定位为`首贷申请评分卡`。模型训练仅使用用户在申请时可以直接填写或平台可以稳定采集的静态申请信息，不使用外部征信历史、不使用站内历史行为变量，也不使用共同借款人/担保方式字段。

本报告的目的，是明确说明以下内容：

- 当前保留的训练字段有哪些
- 每个训练字段在 `South German Credit` 数据集中有哪些取值
- 每个取值分别代表什么业务含义
- 用户申请时需要填写什么数据
- 用户输入值如何映射到训练字段

这样可以保证训练口径和未来线上申请口径一致，避免出现“训练时有、上线时拿不到”的变量错位问题。

---

## 2. 模型标签定义

训练标签来自 `South German Credit` 原始字段：

- 原始字段名：`kredit` / `credit_risk`

原始含义如下：

- `0 = bad`
- `1 = good`

为了后续建模、分箱、WOE 和评分卡解释统一，建议在训练时转换为：

- `bad = 1`：坏客户
- `bad = 0`：好客户

映射规则如下：

- 如果原始 `credit_risk = 0`，则训练标签 `bad = 1`
- 如果原始 `credit_risk = 1`，则训练标签 `bad = 0`

这样做的原因是：

- 评分卡通常围绕“坏样本概率”来建模
- 坏账率、WOE、KS、AUC 的解释更顺手
- 后续做分数映射时逻辑更一致

---

## 3. 当前保留的训练字段

当前确认保留的训练字段共 9 个：

1. 年龄
2. 申请金额
3. 贷款期限
4. 借款用途
5. 居住情况
6. 工作年限
7. 职业类型/职业层级
8. 家庭负担人数
9. 是否有房产/车辆等资产证明

已明确删除：

- 是否有共同借款人或担保方式 `other_debtors`
- `credit_history`
- `number_credits`
- 任何站内历史行为变量

删除原因：

- 当前阶段拿不到外部平台信用历史
- 当前还没有站内历史行为数据
- `other_debtors` 不作为你的申请表核心字段
- 训练字段必须服从上线时可稳定获取的字段

## 3.1 最终保留字段清单

结合当前业务目标、公开数据集可用字段以及未来上线可得性，当前项目的字段体系分为两层：

- 第一层：申请基础字段
- 第二层：历史贷款相关字段

### 3.1.1 申请基础字段

这部分字段是新用户和老用户都可以使用的核心字段，也是申请表中必须采集的字段。

| 序号 | 业务字段 | 数据集原始字段 | 说明 |
|---|---|---|---|
| 1 | 年龄 | `alter` / `age` | 用户实际年龄 |
| 2 | 申请金额 | `hoehe` / `amount` | 本次申请贷款金额 |
| 3 | 贷款期限 | `laufzeit` / `duration` | 本次贷款期限 |
| 4 | 借款用途 | `verw` / `purpose` | 借款的主要用途 |
| 5 | 居住情况 | `wohn` / `housing` | 自有住房、租房或免费居住 |
| 6 | 工作年限 | `beszeit` / `employment_duration` | 当前工作持续时长 |
| 7 | 职业类型/职业层级 | `beruf` / `job` | 职业稳定性和职业层级 |
| 8 | 家庭负担人数 | `pers` / `people_liable` | 需承担的家庭经济负担人数 |
| 9 | 是否有房产/车辆等资产证明 | `verm` / `property` | 可证明的资产基础情况 |

### 3.1.2 历史贷款相关字段

这部分字段在当前原型训练阶段保留，但新用户首贷时不参与实际得分。

| 序号 | 业务字段 | 数据集原始字段 | 说明 |
|---|---|---|---|
| 10 | 历史贷款情况 | `moral` / `credit_history` | 申请人的历史借贷和还款表现 |
| 11 | 历史信贷笔数 | `bishkred` / `number_credits` | 申请人的历史信贷数量 |

### 3.1.3 当前不保留字段

当前不纳入第一版模型或申请表的字段包括：

| 字段 | 原始字段 | 不保留原因 |
|---|---|---|
| 是否有共同借款人或担保方式 | `buerge` / `other_debtors` | 当前不作为申请表核心字段 |
| 外部信用历史 | 数据集中的 `credit_history` 仅作为代理变量 | 真实业务中暂时拿不到其他平台历史 |
| 外部信贷数量 | 数据集中的 `number_credits` 仅作为代理变量 | 真实业务中暂时拿不到外部信贷笔数 |
| 站内历史行为变量 | 无 | 当前仍处于冷启动阶段，尚未积累 |

## 5. 各字段详细说明

### 5.1 年龄

训练字段：

- 原始字段名：`alter` / `age`
- 字段类型：数值型

训练值：

- 为实际年龄整数，例如 `21`、`36`、`48`

训练含义：

- 表示申请人的年龄

用户申请输入：

- 前端问题：`您的年龄是？`
- 输入方式：数值输入
- 示例：`25`

映射方式：

- 用户输入年龄后，直接映射到训练字段 `age`

映射示例：

- 用户填 `25` -> 训练值 `25`
- 用户填 `42` -> 训练值 `42`

备注：

- 年龄是直接映射字段，不需要类别转换
- 真正建模时通常会再做分箱，例如按年龄段分组

---

### 5.2 申请金额

训练字段：

- 原始字段名：`hoehe` / `amount`
- 字段类型：数值型

训练值：

- 为实际贷款金额整数，例如 `1049`、`2799`、`2241`

训练含义：

- 表示本次贷款申请金额

用户申请输入：

- 前端问题：`您本次申请金额是多少？`
- 输入方式：数值输入
- 示例：`5000`

映射方式：

- 用户输入金额后，直接映射到训练字段 `amount`

映射示例：

- 用户填 `3000` -> 训练值 `3000`
- 用户填 `10000` -> 训练值 `10000`

备注：

- 金额是直接映射字段
- 建模时通常按金额区间做分箱

---

### 5.3 贷款期限

训练字段：

- 原始字段名：`laufzeit` / `duration`
- 字段类型：数值型

训练值：

- 为贷款期限数值，例如 `6`、`12`、`18`、`24`

训练含义：

- 表示贷款的还款期限

用户申请输入：

- 前端问题：`您希望贷款期限是多久？`
- 输入方式：枚举选择
- 推荐前端选项：
  - `6个月`
  - `12个月`
  - `18个月`
  - `24个月`

映射方式：

- 用户选择的期限直接映射到训练字段 `duration`

映射示例：

- `6个月` -> `6`
- `12个月` -> `12`
- `24个月` -> `24`

备注：

- 如果以后前端展示为“3期/6期/12期”，需要事先统一口径，确保训练和上线的单位一致

---

### 5.4 借款用途

训练字段：

- 原始字段名：`verw` / `purpose`
- 字段类型：类别型

训练值及含义：

| 训练值 | 含义 |
|---|---|
| `0` | others，其他用途 |
| `1` | car (new)，购买新车 |
| `2` | car (used)，购买二手车 |
| `3` | furniture/equipment，家具/设备 |
| `4` | radio/television，家电/电子产品 |
| `5` | domestic appliances，家庭电器 |
| `6` | repairs，维修 |
| `7` | education，教育 |
| `8` | vacation，旅游/度假 |
| `9` | retraining，再培训 |
| `10` | business，经营/商业用途 |

训练含义：

- 表示借款资金的主要用途

用户申请输入：

- 前端问题：`本次借款主要用途是什么？`
- 推荐前端选项：
  - `购车`
  - `家装/家具/家电`
  - `教育培训`
  - `维修支出`
  - `经营周转`
  - `旅游消费`
  - `其他`

映射方式建议：

- `购车` -> 建议第一版统一映射到 `1`
- `家装/家具/家电` -> 建议第一版统一映射到 `3`
- `教育培训` -> 映射到 `7`
- `维修支出` -> 映射到 `6`
- `经营周转` -> 映射到 `10`
- `旅游消费` -> 映射到 `8`
- `其他` -> 映射到 `0`

备注：

- `purpose` 是一个需要人工定义映射规则的字段
- 第一版不建议把前端用途做得太细，否则映射会很碎
- 最好在前端和训练端保持一张固定映射表

### 5.6 工作年限

训练字段：

- 原始字段名：`beszeit` / `employment_duration`
- 字段类型：有序类别型

训练值及含义：

| 训练值 | 含义 |
|---|---|
| `1` | unemployed，失业/无工作 |
| `2` | < 1 yr，工作未满 1 年 |
| `3` | 1 <= ... < 4 yrs，工作 1 到 3 年 |
| `4` | 4 <= ... < 7 yrs，工作 4 到 6 年 |
| `5` | >= 7 yrs，工作 7 年以上 |

训练含义：

- 表示申请人当前就业持续时间和稳定性

用户申请输入：

- 前端问题：`您当前工作已持续多久？`
- 推荐前端选项：
  - `待业/无固定工作`
  - `未满1年`
  - `1到3年`
  - `4到6年`
  - `7年以上`

映射方式：

- `待业/无固定工作` -> `1`
- `未满1年` -> `2`
- `1到3年` -> `3`
- `4到6年` -> `4`
- `7年以上` -> `5`

备注：

- 这个字段和原始数据集高度一致，适合直接保留

训练含义：

- 表示申请人的职业稳定程度和职业层级

用户申请输入：

- 前端问题：`您目前的职业情况最接近以下哪一项？`
- 推荐前端选项：

映射方式：

- `待业/无固定职业` -> `1`

## 7. 需要特别注意的映射风险

以下字段在使用时要特别注意：

### 7.1 借款用途 `purpose`

- 这是最需要人为制定映射规则的字段
- 前端选项不要无限扩展，否则训练集无法稳定承接
- 建议提前固定一张“前端用途 -> 训练值”的映射表

### 7.2 职业类型/职业层级 `job`

- 第一版最好与训练变量的 4 档职业层级保持一致
- 不建议先做很细的现代职业分类，再强行压回训练数据

### 7.3 家庭负担人数 `people_liable`

- 原始编码和直觉顺序不完全一致
- 建模时不要按数值大小理解好坏
- 只按类别语义理解即可

### 7.4 资产情况 `property`

- 建议保留分层，不要过早合并成简单二分类
- 资产层级通常对评分卡有更好的解释力

## 8. 结论

当前这 9 个字段已经足够支撑一张基础的`首贷申请评分卡`。它们同时满足以下三个条件：

- 训练时可从 `South German Credit` 中找到对应字段
- 业务上有明确含义
- 用户申请时可以稳定填写，并能映射回训练值

因此，后续训练时可以直接以这 9 个字段为核心输入，统一完成：

- 字段抽取
- 标签转换
- 单变量分析
- 分箱
- WOE 编码
- 逻辑回归训练
- 分数卡生成

这意味着，当前字段方案已经可以作为后续评分卡训练和申请表设计的统一基础版本。

## 8.1 训练阶段字段与新用户上线评分字段的区别说明

为了兼顾当前公开数据集训练的可行性和未来真实业务上线的合理性，本项目在字段使用上区分为两个阶段：

- 训练阶段字段
- 新用户上线评分字段

### 8.1.1 训练阶段字段

训练阶段使用的数据字段包括两部分：

- 9 个申请基础字段
- 2 个历史贷款相关字段

也就是说，当前原型训练时，模型总共可使用 11 个字段：

1. 年龄
2. 申请金额
3. 贷款期限
4. 借款用途
5. 居住情况
6. 工作年限
7. 职业类型/职业层级
8. 家庭负担人数
9. 是否有房产/车辆等资产证明
10. 历史贷款情况 `credit_history`
11. 历史信贷笔数 `number_credits`

训练阶段保留这 11 个字段的原因是：

- 当前公开数据集中本身包含这两项历史字段
- 可以先训练出一版结构更完整的原型评分卡
- 便于观察历史贷款字段在公开数据中的区分能力
- 为后续模型扩展预留空间

### 8.1.2 新用户上线评分字段

新用户在首次申请贷款时，平台通常还没有其真实历史借贷记录，因此新用户上线评分时只实际使用 9 个申请基础字段：

1. 年龄
2. 申请金额
3. 贷款期限
4. 借款用途
5. 居住情况
6. 工作年限
7. 职业类型/职业层级
8. 家庭负担人数
9. 是否有房产/车辆等资产证明

对于训练阶段保留的两项历史贷款字段：

- `credit_history`
- `number_credits`

在新用户首贷评分时，直接记为 `0` 分贡献，即：

- 不加分
- 不减分

需要特别说明的是：

- 这里的 `0` 分是评分规则层面的处理
- 并不表示字段原始取值被设成 `0`
- 也不表示新用户“有历史且历史为某种状态”
- 而是表示该用户当前没有可用历史贷款信息，因此该模块暂不参与得分

### 8.1.3 两者区别总结

| 对比项 | 训练阶段 | 新用户上线评分阶段 |
|---|---|---|
| 使用字段数量 | 11 个 | 9 个实际计分字段 |
| 是否使用 `credit_history` | 使用 | 不参与得分，记 0 分 |
| 是否使用 `number_credits` | 使用 | 不参与得分，记 0 分 |
| 历史字段作用 | 提升原型模型完整性 | 暂不生效 |
| 评分结果特点 | 更接近数据集完整信息 | 更贴近真实新用户业务场景 |

### 8.1.4 为什么要这样设计

这样区分训练阶段和上线阶段，主要是为了平衡两个目标：

- 利用公开数据集尽可能训练出完整原型
- 保证新用户上线评分逻辑符合真实业务可得性

如果一开始完全去掉历史字段，虽然更贴近新用户场景，但会失去公开数据集中已有的历史贷款信息；如果一开始把历史字段强行用于新用户评分，又会与真实业务不符。因此，当前方案采用折中设计：

- 训练阶段保留
- 新用户评分阶段中性处理
- 后续再结合老用户真实数据重训练

### 8.1.5 后续演进方向

随着平台逐渐积累用户真实借贷历史，后续可进一步演进为：

- 新用户：仅使用申请基础字段
- 老用户：使用申请基础字段 + 真实历史贷款行为字段

届时可以选择：

- 保留现有评分结构，并把历史贷款分模块替换成平台真实历史变量
- 或者单独训练一版老用户复贷评分模型

这意味着，当前的字段设计既能支持短期原型实现，也能支持后续真实业务升级。

## 9. 历史贷款字段在新用户和老用户场景下的使用方案

虽然当前首贷申请评分卡的核心申请字段是上述 9 个字段，但在原型训练阶段，仍可以保留数据集中的两项历史贷款相关字段参与训练：

- `credit_history`
- `number_credits`

这样做的目的不是因为新用户在申请时一定能提供这两项信息，而是为了在当前公开数据集上先训练出一版包含历史贷款信息的完整原型评分卡。后续当平台积累了自己的真实老用户借贷数据后，再逐步重训练更加贴合真实业务的模型。

### 9.1 历史贷款相关字段

#### 9.1.1 `credit_history`

- 原始字段名：`moral`
- 英文字段名：`credit_history`

训练值及含义：

| 训练值 | 含义 |
|---|---|
| `0` | 过去有过逾期拖欠 |
| `1` | 关键账户异常，或在其他机构有问题信贷 |
| `2` | 以前没借过，或以前贷款都正常结清 |
| `3` | 当前已有贷款，至今还款正常 |
| `4` | 在本机构历史贷款都按时还清 |

字段作用：

- 描述申请人的历史借贷和还款表现
- 反映其过去信用行为质量

#### 9.1.2 `number_credits`

- 原始字段名：`bishkred`
- 英文字段名：`number_credits`

训练值及含义：

| 训练值 | 含义 |
|---|---|
| `1` | 1 笔信贷 |
| `2` | 2 到 3 笔信贷 |
| `3` | 4 到 5 笔信贷 |
| `4` | 6 笔及以上信贷 |

字段作用：

- 描述申请人的历史信贷使用数量
- 反映其借贷经历深度

### 9.2 当前训练阶段的使用原则

当前初次训练时，可以使用数据集中的全部选定特征进行训练，其中包括：

- 9 个申请字段
- `credit_history`
- `number_credits`

也就是说，训练原型模型时，上述两项历史贷款相关字段可以正常参与建模、分箱、WOE 编码和评分卡生成。

这样做的原因是：

- 可以先利用数据集已有信息训练出一版结构更完整的评分卡
- 可以先观察历史贷款相关字段在公开数据中的区分能力
- 为后续“申请特征 + 历史行为特征”的扩展保留设计接口

### 9.3 新用户首贷时的评分规则

对于平台新用户，在首次贷款申请时，通常没有平台内历史借贷记录，因此：

- `credit_history` 不参与实际得分计算
- `number_credits` 不参与实际得分计算

对应评分规则为：

- 这两个字段在新用户评分时直接记为 `0` 分
- 即不加分，也不减分

这里需要特别强调：

- 这并不表示把字段原始取值设为 `0`
- 而是指这两个字段在评分卡最终得分中的贡献值记为 `0`

也就是说，评分时应在规则层面跳过这两项，而不是伪造字段取值。

推荐表达方式：

- `申请基础分`：由 9 个申请字段计算得到
- `历史贷款分`：由 `credit_history` 和 `number_credits` 计算得到
- `最终分 = 申请基础分 + 历史贷款分`

其中：

- 新用户首贷时：`历史贷款分 = 0`
- 因此新用户最终分仅由申请基础分决定

### 9.4 老用户复贷时的评分规则

当平台后续积累了老用户真实借贷数据后，在老用户第二次或后续贷款时，可以将历史贷款信息重新纳入评分。

此时可以有两种演进方向：

#### 方向一：保留当前评分结构

- 新用户：只计算申请基础分
- 老用户：计算申请基础分 + 历史贷款分

这种方式下，历史贷款分可以逐步由平台内真实历史行为替代。

#### 方向二：重训练老用户模型

- 使用平台内老用户真实历史数据
- 重训练一版复贷评分模型
- 将 `credit_history`、`number_credits` 替换成更贴近平台实际业务的历史变量

例如未来可替换成：

- 是否曾经借款成功
- 是否曾经按时还款
- 是否曾经逾期
- 历史最大逾期天数
- 历史借款次数
- 历史结清次数

### 9.5 为什么这是过渡性方案

需要明确说明的是：

- 当前公开数据集中的 `credit_history` 和 `number_credits` 并不等于未来平台自己的真实历史变量
- 因此，这两项在当前阶段属于原型建模中的历史贷款代理变量
- 新用户评分时将其记为 `0` 分，是一种中性处理策略

这意味着：

- 当前模型可以作为过渡版评分卡原型
- 但它不是最终生产版模型
- 后续必须结合真实平台数据进一步重训练和校准

### 9.6 本阶段建议

综合当前业务阶段，建议采用以下方案：

1. 初次训练时，保留 `credit_history` 和 `number_credits` 参与训练。
2. 新用户首贷时，这两项不参与得分，直接记为 `0` 分。
3. 老用户复贷时，再逐步引入真实历史贷款表现。
4. 当平台积累足够老用户样本后，重训练复贷模型或综合模型。

这样既能利用现有公开数据完成原型评分卡训练，又能保证新用户上线时的评分逻辑清晰、可解释，并为后续模型升级预留空间。
